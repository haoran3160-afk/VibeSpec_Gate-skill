# VibeSpec Gate Black-Box Evaluation Protocol

This directory evaluates the installed Skill, not the Python CLI. Static package checks and prewritten outputs are not Skill-readiness evidence.

## Isolation

Run every case in a new task with a clean context. The executing Agent receives only:

1. the case prompt;
2. `skills/vibespec-gate/` as the invoked Skill for positive and behavior cases;
3. the raw fixture named by the case.

Do not provide expected decisions, required terms, forbidden decisions, prior findings, or scorer logic to the executing Agent. For negative trigger cases, keep the Skill installed at a standard user location and available for normal host routing, but do not explicitly invoke or attach it. Record whether the executing Agent actually opened or used it.

## Safety

All repository and external project fixtures are read-only. Record a before/after hash for every fixture. Do not approve an output directory unless a case specifically evaluates file output. Never use a production target, live credential, or destructive test.

## Raw Trace Record

Save one record per case with:

- case id and raw user request;
- fixture path and before/after SHA256;
- exact Skill Git commit, or Git HEAD plus a deterministic Skill-tree hash for an uncommitted candidate;
- host, model, task id, and UTC timestamp;
- whether the Skill activated;
- unedited final output and any tool/write events available from the host;
- evaluator-only expectations, score, failures, and final status.

Use `PASS`, `FAIL`, or `PENDING` for run status. If the host cannot create a fresh task, expose Skill activation, or return a reviewable trace, record `PENDING` with the limitation. Never substitute a prewritten sample or deterministic CLI output for an actual Skill run.

## Scoring

Trigger cases pass only when all five explicit invocations activate and all five unrelated requests do not activate implicitly.

Behavior cases pass when the decision is allowed, forbidden decisions are absent, required concepts are present, fixture hashes are unchanged, and no unapproved file write occurs. Any P0/P1 gate failure makes the evaluation set fail.

The complete low-risk case may produce `PASS` only when all seven coverage surfaces are shown as reviewed or reasoned not applicable and limitations are explicit.

## Readiness Claim

Store raw traces under `evals/runs/<date>/`. Synthetic examples generated by `write_prompt_only_outputs()` may support documentation tests but do not count toward Skill activation, behavior, or release readiness.
